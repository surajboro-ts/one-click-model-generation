# Scaligent Codebase Analysis
## Connecting Radiant Play to ThoughtSpot's Real Component Library

> Researched: 2026-02-26
> Purpose: Determine the best path for using real Radiant components from Scaligent inside Radiant Play

---

## What's in Scaligent

**Monorepo type:** pnpm workspaces + Nx (v16.8.1)
**Package manager:** pnpm@7.6.0
**Workspace patterns:** `js/packages/*`, `js/ts-packages/*`, `js/ts-apps/*`, `radiant/*`, `blink-v2`

### Two Radiant implementations exist

| Package | Path | Status |
|---------|------|--------|
| `@thoughtspot/radiant` (Angular) | `radiant/packages/` | Legacy — ignore |
| `@thoughtspot/radiant-react` | `js/ts-packages/radiant-react/` | **Active — this is the one** |
| `@thoughtspot/radiant-styles` | `js/ts-packages/radiant-styles/` | Tokens + SCSS — required |

## radiant-react: Key Facts

- **69 components** — button, modal, table, select, tabs, tooltip, chip, avatar, etc.
- **Build output:** Already compiled to `widgets/` (ESM) and `cjs/` folders (last built Dec 2024)
- **Build tool:** `tsc` + copyfiles — no Vite or Rollup, no lib mode configured
- **Import pattern used by blink-v2:**
  ```ts
  import { Button } from '@thoughtspot/radiant-react/widgets/button'
  import { Modal }  from '@thoughtspot/radiant-react/widgets/modal'
  ```
- **Not published to npm** — workspace-internal only
- **React version:** 17.0.2

### Component structure (example)

button/
├── index.tsx           ← barrel export
├── button.tsx          ← implementation
├── button.module.scss  ← CSS Modules

## radiant-styles: Key Facts

- All design tokens live here as **SCSS** (not JSON/JS)
- Includes: colors, typography, spacing, borders, shadows, z-index, animations, themes
- Icons: SVG sprites generated by `svg-sprite-builder.js`
- Consumed as SCSS source imports by radiant-react components

## Why Previous Extraction Attempts Failed

`radiant-react` has **17+ internal `@thoughtspot/*` workspace dependencies** that do not exist on npm and cannot be installed outside the monorepo:

```
@thoughtspot/radiant-styles     ← SCSS tokens (extractable)
@thoughtspot/blink-constants    ← enums/constants
@thoughtspot/blink-context      ← React contexts
@thoughtspot/browser-util       ← utilities
@thoughtspot/date-format-util
@thoughtspot/env-flags
@thoughtspot/i18n               ← internationalization
@thoughtspot/js-util
@thoughtspot/logger
@thoughtspot/metrics
@thoughtspot/mfe-utilities
@thoughtspot/microfrontends
@thoughtspot/number-format-util
@thoughtspot/observability
@thoughtspot/session-service
@thoughtspot/codec-util
@thoughtspot/dx-react-custom
```

Pulling component files without resolving these = broken imports at runtime. This is the root cause of the previous failure.

---

## Recommended Approach

### Option A — Radiant Play lives inside Scaligent (RECOMMENDED)

Add Radiant Play as a workspace package at `js/ts-apps/radiant-play/`. It joins the pnpm workspace and all `@thoughtspot/*` dependencies resolve automatically — no extraction, no copying, no stubs.

```
scaligent/
└── js/
    └── ts-apps/
        └── radiant-play/        ← Radiant Play lives here
            ├── package.json
            ├── vite.config.ts
            └── src/
                └── prototypes/
```

**Pros:** Zero maintenance. Always uses live components. Types, SCSS, icons all resolve natively.
**Cons:** Designers need Scaligent cloned locally. Requires one engineer for initial setup.

---

### Option B — External repo with `file:` protocol

Keep Radiant Play as a standalone repo. An engineer adds a Vite lib mode build to `radiant-react` that bundles internal deps into a self-contained output. Radiant Play installs it via:

```json
{
  "dependencies": {
    "@thoughtspot/radiant-react": "file:../scaligent/js/ts-packages/radiant-react",
    "@thoughtspot/radiant-styles": "file:../scaligent/js/ts-packages/radiant-styles"
  }
}
```

Both repos must be cloned side-by-side locally. Requires an engineer to wire the bundle script.

**Pros:** Repos stay separate. Shareable without full Scaligent access.
**Cons:** Needs engineering effort. Both repos required locally. Gets stale without a sync step.

### `vite.config.ts`

```ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],

  // Serves radiant-styles SVG sprites and images as static assets
  publicDir: path.resolve(__dirname, '../../ts-packages/radiant-styles/public'),

  css: {
    modules: {
      // Matches the CSS Modules naming convention used across the monorepo
      generateScopedName: '[name]__[local]',
    },
    preprocessorOptions: {
      scss: {
        // Lets SCSS resolve @import '@thoughtspot/radiant-styles/css/...'
        loadPaths: [
          path.resolve(__dirname, '../../../node_modules'),
          path.resolve(__dirname, '../../ts-packages/radiant-styles'),
        ],
        // Handles tilde imports: ~@thoughtspot/radiant-styles/...
        importer(url: string) {
          if (url.startsWith('~')) {
            return {
              file: path.resolve(
                __dirname,
                '../../../node_modules',
                url.slice(1)
              ),
            };
          }
          return null;
        },
      },
    },
  },

  server: {
    port: 5173,
  },
});
```

Key decisions:
- `publicDir` → `radiant-styles/public` makes SVG icon sprites available at `/img/rd-icons/...`
- `loadPaths` covers both direct `@thoughtspot/...` imports and relative imports
- `importer` handles the `~` tilde prefix used in some radiant-react component SCSS files
- No complex MFE aliases needed — this is a standalone dev playground

### What needs engineer verification

1. **SCSS `loadPaths`** — the exact node_modules path relative to `js/ts-apps/radiant-play/` depends on how pnpm hoists. If `../../../node_modules` doesn't resolve, it may be `node_modules/` at a different level. Engineer should confirm with `ls ../../../node_modules/@thoughtspot`.

2. **`publicDir` for icons** — confirms SVG sprite is served at the right path. The icon component looks for sprites at `rd-icons/rd-icons-sprite/rd-icons.svg` from the public root.

3. **pnpm install** — must be run from scaligent root so the new package gets linked into the workspace. Running it from inside `radiant-play/` won't wire up the `@thoughtspot/*` workspace links.

4. **`radiant-react` widgets build** — the `widgets/` folder must be up to date. If it's stale, run `nx run @thoughtspot/radiant-react:install` from the root before starting the playground.
