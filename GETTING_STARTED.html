01 · What is Radiant Play?

A playground where
prototypes become real.

Radiant Play is a sandbox built for ThoughtSpot designers to create interactive prototypes and interactions using the actual Radiant design system components — the same ones that ship in the product. No rebuilding. No translation layer.

Without Radiant Play
🎨  Figma file — looks perfect
📐  Developer re-implements everything
🔄  Spec review back-and-forth
🐛  "That's not quite right" bugs
⏳  Weeks of iteration...
Design intent gets lost in translation.
▶ With Radiant Play
✦  Describe your design to AI in Cursor
🧩  AI builds with real Radiant components
👀  Live interactive preview instantly
🔀  Merge → auto-deploys to Vercel
🚀  Developers use your prototype directly
The playground is the handoff. Play → ship.
02 · Key Concepts

5 words. That's all
you need to know.

The playground runs on Git — but you only need five terms to use it comfortably. Each one maps directly to something you already do in Figma. Tap any card to see the analogy.

📁
Repository

The entire project folder, tracked by Git. Everyone works from the same repo.

Tap to see Figma analogy →
In Figma terms
= A shared Figma file.

Everyone can see it, fork it to their own drafts, and propose changes back. The repo is the single source of truth — just like your team's main Figma library.
🌿
Branch

A parallel copy of the project where you work without affecting anyone else.

Tap to see Figma analogy →
In Figma terms
= A new Figma page for experiments.

You add a page to try an idea. Main design is untouched. When it's ready, you move the work back. That's a branch.
📷
Commit

A named snapshot of your changes. Like saving a version in Figma.

Tap to see Figma analogy →
In Figma terms
= File → Save to Version History.

Every commit has a name, timestamp, and author. You can jump back to any commit — like restoring a named version.
☁️
Push / Pull

Sync your work to and from GitHub — upload or download changes.

Tap to see Figma analogy →
In Figma terms
Push = save to cloud.
Pull = refresh from cloud.

Like syncing a Google Drive folder. Push uploads your commits; Pull downloads what your teammates pushed.
🔀
Pull Request

A request to merge your branch into main — reviewed by a teammate before going live.

Tap to see Figma analogy →
In Figma terms
= Tagging a teammate for design review.

"Hey @faris, can you review my changes before they go to main?" A PR is that review — but for code. Approved → merged → auto-deployed.
03 · Branches

Play freely.
Nothing breaks.

Every prototype you build lives on its own branch — an isolated copy of the playground that only you can see. Experiment, break things, start over. The live site on main is always safe until you decide to merge.

main branch
The published, live version at radiantplay.vercel.app. Always stable. Only changes after a reviewed Pull Request is merged.
your branch
Your personal workspace. Invisible to everyone until you open a PR. Build, break, experiment — main is never affected.
🎮
Name your branch like it describes the work

Use lowercase letters and hyphens. Good examples: add-muse-chat-prototype  ·  fix-sidebar-overlap  ·  experiment-dark-mode

06 · Setup

Get the playground
running in 5 minutes.

Set this up once and you're in — after that it's just npm run dev and you're back in the playground.

⬡
Node.js 18+ Required

The JavaScript runtime. Download from nodejs.org — just click install, no configuration needed.

✦
Cursor IDE Required

Code editor with AI built in. Where you'll describe designs and preview prototypes. Download from cursor.so.

🐙
GitHub Account Required

Free at github.com. Where your code lives and where you open Pull Requests for team review.

🖥️
GitHub Desktop Optional

Visual Git interface — no terminal required. Great if command lines feel uncomfortable. desktop.github.com.

1
Fork the repository on GitHub

On the project's GitHub page, click the Fork button (top-right corner). This creates your own copy — like duplicating a Figma file to your drafts.

2
Clone your fork to your computer

Open Cursor, press Ctrl+` to open the terminal, then run:

git clone https://github.com/YOUR-USERNAME/radiantplay.git
3
Open the folder in Cursor

In Cursor: File → Open Folder → select the radiantplay folder you just cloned.

4
Install dependencies

Downloads all Radiant component libraries. Takes about a minute.

npm install
5
Start the dev server

Run this then open http://localhost:5173 — you'll see the Radiant Play homepage.

npm run dev
🎉
You're set up!

Next time just open Cursor and run npm run dev. Never need to reinstall unless the project's dependencies change.

07 · Building in the Playground

Describe it.
Play with it. Ship it.

Open Cursor's AI chat with Cmd+L, describe your interaction or screen, and the AI assembles it from real Radiant components. Tweak, iterate, and watch it come alive in the browser.

🤖
How to talk to the AI

Describe your design like you'd write a Figma annotation — layout, interactions, states, all of it. Example: "Build a settings page with a sidebar for General, Notifications, and Security. The main panel has Name and Email inputs, notification toggles, and a Save button. Use Radiant components and tokens." Or paste a Figma screenshot and say: "Recreate this interaction using Radiant components."

1
Create a branch
git checkout -b add-my-first-prototype
2
Scaffold the prototype folder

This creates the folder structure automatically — index.tsx, components/, and a README.md for your design notes.

npm run new-prototype MyFirstPrototype
3
Open the file in Cursor, press Cmd+L, and describe your design

Open src/prototypes/MyFirstPrototype/index.tsx. The AI reads the project context and knows which Radiant components are available. Just describe what you want.

4
Preview at localhost:5173/playground

Your prototype appears in the gallery automatically. The page hot-reloads instantly as you save changes.

5
Commit and push when ready
git add . git commit -m "add settings page prototype with sidebar and form" git push -u origin add-my-first-prototype

git checkout main && git pull

09 · FAQ

Questions every
designer asks.

Honest answers for designers getting started in the Radiant Play playground.

Git is a tool that lives on your computer and tracks every change you make to files. Think of it like Time Machine for your project folder.

GitHub is a website where your project is stored online so you can share it with others and access it from anywhere. Git is the engine; GitHub is the garage where everyone parks.

No. GitHub Desktop gives you a visual interface for every Git operation — creating branches, committing, pushing, pulling, and merging — with zero terminal required. Download it free from desktop.github.com.

That said, four terminal commands make you significantly faster: git checkout -b, git add ., git commit -m, git push. Cursor has a built-in terminal — open it with Ctrl+`.

That's exactly what branches are for. Main is always safe. If your branch is a disaster, delete it and start fresh with zero consequences:

git checkout main git branch -D my-broken-branch git checkout -b my-fresh-start

You can also jump back to any earlier commit — every commit is a save point you can restore.

Conceptually the workflow is very similar. Your branch = a Figma exploration page. Committing = saving a named version. Merging = moving that exploration into the delivered file. A Pull Request = asking a teammate to review before publishing.

The key difference: the output is real, interactive code using actual Radiant components. Developers can fork your prototype and ship it — no re-implementation needed. That's the whole point.

origin is just a nickname Git automatically sets for your GitHub repository's URL. When you cloned the project, Git saved the full URL as "origin" so you never have to type it. git push origin just means "upload to my GitHub repo."

Commit whenever you finish a small, self-contained piece of work. Think of it like saving checkpoints in a game. Five small commits is far better than one giant commit at the end — each one is a save point you can return to if something breaks.

Good cadence: scaffold → layout → content → styling → interactions → done.
