# Banners

### **Definition**

System level banner alerts inform the user of a critically important event that affects the entire system. They are intended to be noticed and acted upon by the user. They are color coded for different levels of severity.

Section alerts inform the user that something that has happened in a specific section of the screen. They appear above the affected area and follow the same severity color code as the system alerts.

### **Usage**

Communicate a state that affects the entire system, not just a feature or page. It persists over a session and appears without the user initiating the action.

- Provide primary and secondary actions for the user: Fix the issue and Report the issue
- Prefer global alerts for most system-wide events, reserving error alerts exclusively for critical errors.
- Provide a close button when the system has caused the error.
- Avoid using global alerts for cases that aren’t urgent and won’t disrupt workflow or change the user experience.
- Don’t provide more than two action buttons.
- Don’t provide a close button when the user has caused the error.

---

### **Content guidelines**

- Alert banner messaging should be clear and efficient
- Use either a short phrase or phrases that describes what’s happening and what the user needs to know
- When possible, provide an action the user can take to remedy the situation.
- Do not include periods for short phrases or single sentences
- See the [alert content guidelines](https://radiant.thoughtspot.com/4be2fa497/p/44aeab-writing-for-components/t/0305da)
