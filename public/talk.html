01
Token architecture first
Started by building out the token layer — color, spacing, typography — before any components were touched
02
Bringing initial components from Figma
Mapped and extracted the initial set of components from Figma designs, converting them into code-based, usable building blocks
03
Connecting with the ThoughtSpot codebase
Linked to the ThoughtSpot codebase to pull production-ready components directly from source, so they stay current
This was necessary because our design system was not readily available, updated, or modern enough at the time. RadiantPlay was created as a more sophisticated and advanced layer that designers could actually work with.
Section 3. Orchestrator

The AI pipeline

Every prompt flows through 5 steps, from description to working React code.

01  Your prompt                      # text, Figma screenshot, or context
      │
      ▼
02  Orchestrator classifies          # assigns Tier 0–3, picks rule files
      │
      ▼
03  Loads relevant rules only        # up to 10 files, on demand
      │
      ▼
04  Pre-implementation gate          # 4 checks before writing code
      │
      ▼
05  Generates React + tokens         # output already follows the system
Intent classification

Tier 0–3 routing

The orchestrator assigns every task a tier. Higher tiers load more context, but most tasks stay at Tier 0 or 1.

TIER 0
Instant
Answered from memory. No rule files loaded.
Component exists? Token name?
TIER 1
Targeted
1–3 rule files. Single component or small fix.
Add a table row. Change spacing.
TIER 2
Contextual
4–7 files. New section inside a prototype.
Build a settings panel. Add a modal.
TIER 3
Full context
All relevant files. New prototype from scratch.
Build a Liveboard experience.
~60%Context reduction for median tasks. Most daily prompts skip heavy rule-loading entirely, resulting in faster and more accurate outputs.
The instruction system

Reference files

Around 20 rule files in total. Each loads on demand — only what the task needs, when it needs it. Here are the key ones.

▸
CLAUDE.md , always loaded. Core conventions: component rules, token imports, forbidden patterns. ~135 lines.
▸
_orchestration.md , router. Reads intent and decides which other files to load.
▸
component-inventory.md , skill. 73+ components with props, usage, "use X not Y" mappings.
▸
layout-patterns.md , skill. Horizontal, Vertical, Grid, AppShell, correct primitives for every context.
▸
token-usage.md , skill. Color, spacing, typography. Enforces no hardcoded values.
▸
modal-patterns.md / widget-patterns.md , skills. Loaded only when the task needs them.
Quality guardrails

Pre-implementation gate

Before any code is written, the AI runs 4 checks. It also monitors the session for inefficiency patterns.

4 checks before writing code
✓
Component exists? Verify against inventory before creating custom
✓
CSS anti-patterns? Reject hardcoded hex, magic spacing, inline flex
✓
Icon name valid? Confirm it exists in the Radiant icon set
✓
Forbidden words? Block pattern violations before output
Session awareness
▸
3+ single-property tweaks in a row, suggests batching into one prompt
▸
Figma MCP overhead, flags the ~4,000 token cost when idle
▸
Context budget, suggests /compact on topic switches
▸
Session memory, tracks already-loaded rule files
Section 4. Fork + clone model

Why fork, not branch?

Every designer gets their own full fork of RadiantPlay, not a branch. They own it completely.

Problems with a shared branch
×
Merge conflicts on every sync
Designer prototypes collide in the same registry file
×
No isolation
One broken prototype breaks everyone's dev server
×
Unclear ownership
Designer work mixed with platform work, hard to change safely
×
Coordination overhead
Everyone has to sync before they can merge
What fork gives us
✓
Full autonomy
Deploy independently to Vercel, no dependency on others
✓
Clean upstream syncs
Core library updates flow in without touching designer prototypes
✓
Clear ownership
Your fork = your prototypes. Contributing back is a deliberate choice
✓
Scale without coordination
15+ forks run independently, no sync meetings
The technical solution

Registry split

Split the prototype registry into three files so upstream and designer work never collide.

Upstream
registry-core.ts
Maintained by the platform team. Shared templates and reference prototypes. Syncs cleanly via sync-upstream.
Designer
registry-mine.ts
Owned entirely by the designer. Upstream syncs never touch this file. Add, rename, delete freely, zero conflict risk.
Merger
registry.ts
Thin file that imports both and combines them. The app sees one registry. Identical across all forks, never changes.
0Merge conflicts, ever. Before: every sync required manual conflict resolution. After: the split makes them structurally impossible.

The accumulation effect

How patterns compound

Each prototype adds to a shared library of context. The system gets smarter as more work happens inside it.

01 · Designer ships a feature
▸
Builds prototype in their fork
▸
Uses component inventory + context files
▸
Iterates with real data and state
▸
Ships to staging for review
→
02 · Pattern extracted
▸
Context file written for the domain
▸
Shared components pulled into _shared/
▸
Template scaffold committed upstream
▸
Rule file updated with learnings
→
03 · Next designer inherits
▸
Starts from scaffold, not blank page
▸
AI already knows the domain patterns
▸
Faster first prototype, better quality
▸
Adds their own patterns back
Context files document how each product function works. They help the broader team understand prototypes, work within guardrails, and see where rules are breaking. They also make scaling and replication much easier.
Section 6. The common library approach

The recipe

Four ingredients. You probably have most of them already.

01
A component library
Your existing design system, imported as a real package, not duplicated.
Radiant / MUI / Chakra
02
An AI coding tool
Anything that reads files and writes code. Tool-agnostic.
Claude Code / Cursor / Copilot
03
An instruction file
CLAUDE.md or .cursorrules, which components to use, which to avoid.
CLAUDE.md / .cursorrules
04
Fork-per-designer
Each designer forks the base repo. Upstream flows in without conflict.
GitHub / GitLab / any host
